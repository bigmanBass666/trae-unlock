## 2026-04-19

### 新发现: sandbox_execute_failure 弹窗根因

**问题**: AI 执行 python 命令时出现 "命令在沙箱中运行不通过，是否仍然要在沙箱外运行？" 弹窗

**根因分析**:
- `getRunCommandCardBranch` 方法中 WHITELIST 模式有独立分支逻辑
- 当 `block_level = "sandbox_execute_failure"` 时触发 `V2_Sandbox_Execute_Failure` 弹窗
- 原有的 `bypass-runcommandcard-redlist` 补丁只处理了 `ALWAYS_RUN` 和 `default` 模式
- WHITELIST 模式完全没被处理！

**补丁修复**:
- 新增 `bypass-whitelist-sandbox-blocks` 补丁
- 位置: ~8069700
- 效果: WHITELIST 模式下所有沙箱 block 分支都直接返回 `P8.Default`（自动执行）
- 修复后: python 等命令在沙箱模式下应该零弹窗自动执行

**指纹检测修复**:
- 原 `check_fingerprint` 使用了原代码中存在的字符串，导致误判为"已应用"
- 已修改为替换后的唯一字符串 `case Cr.AutoRunMode.WHITELIST:return P8.Default;`

### 架构新认知

| 确认模式 | 原行为 | 补丁后 |
|---------|--------|--------|
| WHITELIST + sandbox blocks | 弹窗确认 | 直接执行 |
| ALWAYS_RUN + RedList | 弹窗确认 | 直接执行 |
| default + RedList | 弹窗确认 | 直接执行 |

## 2026-04-19 上午

### 新功能: 绕过循环检测限制

**问题**: 模型长运行时出现 "检测到模型陷入循环，为避免更多消耗已主动中断对话，建议更换描述后重试" 错误并中断对话

**根因分析**:
- 服务端检测到模型重复工具调用或内容循环时返回错误码
- 错误码: `LLM_STOP_DUP_TOOL_CALL (4000009)` - 重复工具调用循环
- 错误码: `LLM_STOP_CONTENT_LOOP (4000012)` - 内容循环
- 这两个错误码不在"可继续"错误列表(J变量)中 → 只显示错误消息，无"继续"按钮
- 对话被强制中断，用户无法续接

**补丁方案**: `bypass-loop-detection`
- 位置: ~8696378 (J变量定义)
- 原始: `J=!![kg.MODEL_OUTPUT_TOO_LONG,kg.TASK_TURN_EXCEEDED_ERROR].includes(_)`
- 替换为: `J=!![... ,kg.LLM_STOP_DUP_TOOL_CALL,kg.LLM_STOP_CONTENT_LOOP].includes(_)`
- 效果: 循环检测错误被识别为"可继续"错误 → 自动触发 auto-continue-thinking 补丁 → 无感发送"继续"消息 → 模型无缝续接

**工作原理**:
1. 服务端返回 4000009/4000012 错误
2. 客户端 J变量检查通过（因为已加入列表）
3. 触发 if(V&&J) 分支（auto-continue-thinking 已修改此分支）
4. setTimeout(50ms) 自动点击"继续"
5. return null 不显示 Alert 弹窗
6. 模型收到"Continue"消息后无缝继续工作

### 新功能: 服务层确认状态同步更新 (修复切窗口沙箱卡住)

**问题**: 沙箱命令可以自动跑到沙箱外了，但切换到其他会话窗口后又被阻塞，需要重新打开窗口才自动执行

**根因分析**:
- `service-layer-runcommand-confirm` 补丁在服务层调用了 `provideUserResponse({decision: "confirm"})`
- 但**没有更新本地 store 中 planItem 的 `confirm_info.confirm_status`**
- React 组件读取 store 中的状态 → 仍然是 `"unconfirmed"` → 继续显示"等待操作"
- 切换窗口后 React 冻结 → 即使服务端已执行命令，UI 仍卡住

**补丁方案**: `service-layer-confirm-status-update`
- 位置: ~7503400 (service-layer-runcommand-confirm 补丁内)
- 在 `provideUserResponse` 调用后追加: `e.confirm_info&&(e.confirm_info.confirm_status="confirmed")`
- 效果: 服务层确认后立即将 planItem 的确认状态改为 "confirmed"
- 后续 SSE 流处理时，planItem 已经是 confirmed 状态 → React 组件不会显示"等待操作"

**关键架构洞察**:
- 之前以为服务层补丁就够了 → 但忽略了**本地状态同步**
- `updateRunCommandToolStatusFromUser` 中 Confirmed 分支会更新本地 planItem 状态
- 我们的服务层补丁只调了 API，没更新本地状态 → 状态不同步
- 解决方案: 在服务层补丁中也更新 planItem 的 confirm_info

### 新功能: 精准化自动确认过滤 (修复误判用户问答)

**问题**: 自动确认太激进 — AI 向用户提问的选项也被自动"确认"了，用户无法手动选择

**根因分析**:
- `auto-confirm-commands` 和 `service-layer-runcommand-confirm` 两个补丁会自动确认**所有** `confirm_status === "unconfirmed"` 的项目
- 包括 AskUserQuestionCard（`toolName = "response_to_user"`）— 用户问答类型
- 导致 AI 列出的选项全部被自动点击"下一步"/"提交"，用户无法参与选择

**技术发现**: ToolCallName 完整枚举 (位置 ~41400)
| 枚举值 | 字符串值 | 类型 |
|--------|---------|------|
| `RunCommand` | `"run_command"` | 命令执行 ✅ 需要自动确认 |
| `ResponseToUser` | `"response_to_user"` | 用户问答 ❌ 不应自动确认 |
| 其他 30+ 种 | ... | 文件操作/搜索/MCP 等 |

**补丁方案**: v3 精准过滤
- **`service-layer-runcommand-confirm` (v3)**: 增加白名单 `(e?.toolName==="run_command")`
  - 只允许 `run_command` 类型自动确认
  - `response_to_user` 被排除 → AskUserQuestion 正常显示 UI
- **`service-layer-confirm-status-update` (v2)**: 同步增加白名单过滤
- **`auto-confirm-commands` (v2)**: 增加黑名单 `(e?.toolName==="response_to_user")`

**效果**:
- ✅ RunCommandCard → 自动确认（白名单通过）
- ✅ AskUserQuestionCard → 显示 UI 让用户选择（被排除）
- ⚠️ 其他工具类型 → 默认不自动确认（保守策略）
