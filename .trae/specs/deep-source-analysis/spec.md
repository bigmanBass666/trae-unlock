# 源码架构深度解读 Spec

## Why

当前补丁开发模式是"出了问题才找代码"，导致：
- 7次迭代才解决自动确认（不了解完整数据流）
- 聊天窗口崩溃（不了解 this 绑定和错误传播机制）
- 双重调用 bug（不了解两个补丁的交互关系）
- 每次探索都从零开始，无法复用知识

需要一份系统性的架构文档，让后续补丁开发从"盲人摸象"变成"按图索骥"。

## What Changes

- 对 Trae AI 聊天模块的核心子系统做深度源码解读
- 产出结构化的架构文档，包含模块边界、数据流、关键类/方法、状态管理
- 重点标注所有"限制点"（确认弹窗、错误阻断、沙箱拦截等）
- 建立模块间的交互关系图

## Impact

- Affected files: `docs/architecture/source-architecture.md`（大幅扩展）
- 新增文件: `docs/architecture/` 下按子系统拆分的架构文档
- 后续所有补丁开发都将基于此架构文档

## 分析范围

### 第一优先级：核心数据流（直接影响补丁开发）

1. **SSE 流解析系统** — PlanItemStreamParser 完整生命周期
   - 数据入口：SSE 连接建立、消息接收
   - 事件分发：不同 event type 的处理逻辑
   - 状态更新：planItem 的创建、更新、确认流程
   - 错误处理：错误码检测、Alert 渲染、恢复机制

2. **命令确认系统** — 从用户点击到命令执行的完整链路
   - confirm_info 数据结构完整定义
   - provideUserResponse API 调用链
   - 本地状态同步机制（store 更新 → React 渲染）
   - 各 BlockLevel 的判定逻辑和分支

3. **沙箱执行系统** — 命令从 AI 生成到实际执行的全过程
   - 沙箱模式判定（AutoRunMode 枚举）
   - 沙箱外执行请求流程
   - RunCommandCard 组件完整状态机

### 第二优先级：限制点地图（主动发现所有可解锁的限制）

4. **错误码与阻断系统** — 所有会中断对话的错误
   - 完整错误码枚举及含义
   - J 变量（可继续错误列表）的完整逻辑
   - Alert 组件的所有渲染条件和类型

5. **UI 限制系统** — 所有用户可见的限制弹窗/提示
   - 确认弹窗（RunCommandCard、AskUserQuestionCard 等）
   - 错误提示（循环检测、token 耗尽等）
   - 功能限制（思考上限、对话轮次等）

### 第三优先级：基础设施（支撑后续开发）

6. **模块边界与依赖关系** — 整体架构地图
   - 主要类/模块的定义位置和职责
   - 服务注册和依赖注入机制
   - 事件系统（发布/订阅模式）

## ADDED Requirements

### Requirement: 子系统架构文档

每个子系统的架构文档 SHALL 包含以下结构：

1. **概述**：子系统职责、在整体架构中的位置
2. **核心类/方法**：关键类定义位置、方法签名、职责说明
3. **数据流图**：从输入到输出的完整路径，标注关键节点
4. **状态管理**：涉及的状态变量、更新时机、读取方
5. **限制点清单**：所有可被补丁修改的限制点，标注位置和类型
6. **补丁接口**：推荐的补丁注入点，标注安全注意事项

#### Scenario: 补丁开发者使用架构文档
- **WHEN** 开发者需要新增一个补丁
- **THEN** 可以在架构文档中找到目标子系统的完整数据流
- **AND** 确认最佳注入点和潜在风险

### Requirement: 限制点地图

系统 SHALL 提供一份完整的限制点地图，包含：

- 每个限制点的位置（offset）、类型（弹窗/错误/阻断）、触发条件
- 当前是否已有补丁覆盖
- 推荐的解锁方案

#### Scenario: 主动发现新限制
- **WHEN** 用户遇到一个新的限制弹窗
- **THEN** 可以在限制点地图中查找对应条目
- **AND** 了解该限制的完整机制和推荐解锁方案

### Requirement: 分析方法

源码分析 SHALL 采用以下方法：

1. **关键字符串定位**：通过已知字符串（如 "provideUserResponse"）定位核心代码
2. **上下文扩展**：从定位点向前后扩展，理解完整逻辑
3. **调用链追踪**：追踪方法调用和被调用关系
4. **枚举提取**：提取所有枚举定义（错误码、状态、类型等）
5. **数据流还原**：从 SSE 入口到 UI 渲染的完整路径

#### Scenario: 分析 SSE 流解析
- **WHEN** 分析 PlanItemStreamParser
- **THEN** 从 SSE 连接入口开始，追踪每个 event type 的处理逻辑
- **AND** 标注所有状态变更点和 UI 更新触发点
