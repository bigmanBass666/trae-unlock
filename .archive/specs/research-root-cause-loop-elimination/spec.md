# 根源消除循环限制可行性研究 Spec

## Why

当前补丁是"事后补救"——循环检测触发后自动续接。用户希望从根源消除循环限制，这样根本不需要续接。

## 调研结论

### 核心事实：循环检测 100% 在服务端

| 方面 | 服务端 | 客户端 |
|------|--------|--------|
| **检测逻辑** | 计数重复工具调用 / 比较相似内容 | 无 |
| **决策** | 判定是否触发循环 → 返回错误码 | 无 |
| **错误码生成** | 生成 4000009/4000012 | 只接收 |
| **可修改性** | 不可修改 | 可修改响应行为 |

客户端搜索了 30+ 个模式（dupToolCallCount、repeatCount、loopDetect、isLoop、maxRepeat 等），**全部未找到**。客户端零参与循环检测决策。

### 行业参考：Gemini CLI 的 LoopDetectionService

Google Gemini CLI 的循环检测是**客户端实现**的（`LoopDetectionService`），但 Trae 的循环检测是**服务端实现**的。两者架构不同：

| 方面 | Gemini CLI | Trae |
|------|-----------|------|
| 检测位置 | 客户端 | 服务端 |
| 可绕过性 | 高（直接修改客户端代码） | 低（无法修改服务端） |
| 阈值 | TOOL_CALL_LOOP_THRESHOLD=5 | 未知（服务端配置） |
| 禁用方式 | `disabledForSession=true` | 无客户端开关 |

### OpenAI 的 hop counter 机制

OpenAI 使用服务端 hop counter + 唯一请求 ID 来防止无限循环，与 Trae 类似，都是服务端决策。

## 可行性分析

### 方案 A: 修改服务端请求参数 ❌ 不可行

- 循环检测在服务端，我们无法修改服务端代码
- 没有客户端参数可以禁用或调整循环检测阈值

### 方案 B: 伪造/过滤 SSE 流中的错误码 ⚠️ 理论可行但风险极高

- 在 SSE 流解析层拦截 4000009/4000012 错误码
- 将其替换为正常完成信号
- **风险**：服务端可能已经停止生成，替换错误码不会让模型继续输出

### 方案 C: 当前方案（事后自动续接）✅ 已实现且有效

- bypass-loop-detection v3：J 数组扩展
- efh-resume-list v2：可恢复列表扩展
- auto-continue-thinking v3：ec() 自动续接
- **效果**：循环检测触发后自动 resumeChat，对话无缝继续

### 方案 D: 减少循环检测触发频率 ⚠️ 值得探索

- 在客户端修改发送给服务端的请求，添加提示词避免重复工具调用
- 在客户端缓存工具调用结果，避免重复请求
- **局限**：只能减少误触发，不能完全消除

## 结论

**从根源消除循环限制不可行**，因为检测逻辑在服务端。当前方案 C（事后自动续接）是最优解。

但方案 D（减少触发频率）可以作为补充优化。
