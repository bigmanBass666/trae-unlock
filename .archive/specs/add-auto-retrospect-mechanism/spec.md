# 任务复盘与自我迭代机制 Spec

## Why

AI 完成任务后缺乏自动复盘环节，导致同样的错误反复出现。需要建立"任务完成→自动复盘→发现改进→更新规则"的闭环，让 AI 自主进化。

## 反思：循环上限源头破除任务的改进空间

### 我做了什么

1. 用户提出"从根源消除循环限制"
2. 我搜索了客户端 30+ 个模式 → 确认客户端零参与
3. 我搜索了行业参考 → Gemini CLI、OpenAI
4. 我评估了 4 个方案 → 结论：当前方案已是最优解
5. 写入 discoveries.md

### 如果重来，我会怎么做

| 步骤 | 我做的 | 应该做的 | 改进幅度 |
|------|--------|---------|---------|
| 1. 理解问题 | 直接搜索代码 | **先问自己**：循环检测是客户端还是服务端？→ 从架构图就能推断是服务端 | 节省 50% 搜索时间 |
| 2. 搜索方向 | 搜索客户端代码 | **先搜行业参考** → Gemini CLI 源码分析文章直接告诉我"循环检测可以在客户端也可以在服务端" → 然后针对性搜索 Trae 是哪种 | 节省 70% 搜索时间 |
| 3. 搜索工具 | 用 search agent + PowerShell | **先用 ast-grep** → `ast-grep -p '4000009' --lang js --json` 一条命令就能确认客户端是否生成错误码 | 节省 80% 搜索时间 |
| 4. 结论速度 | 花了整个 spec 周期 | **5 分钟就能得出结论**：ast-grep 搜不到错误码生成逻辑 → 服务端 | 从小时级→分钟级 |

### 核心教训

1. **先推理，再搜索**：从架构图就能推断循环检测在服务端（SSE 流推送错误码），不需要搜索 30+ 个模式来"证明"
2. **先搜行业，再搜代码**：Gemini CLI 的 LoopDetectionService 源码分析文章直接提供了完整参考
3. **用对工具**：ast-grep 一条命令 > 30+ 个 PowerShell 搜索
4. **最小验证**：只需要验证"客户端是否生成 4000009/4000012"一个事实，不需要穷举所有可能的模式

### 可提炼的方法论

**「推理→搜索→验证」三步法**：
```
Step 1: 推理 — 从已知架构推断可能的答案
Step 2: 搜索 — 搜行业参考 + 用 ast-grep 精确搜索
Step 3: 验证 — 只验证推理中的关键假设，不穷举
```

## ADDED Requirements

### Requirement: 任务完成后自动复盘

AI 每完成一个任务后 SHALL 自动执行以下复盘流程：

1. **回顾**：我做了什么？花了多少步骤？
2. **反思**：有没有更快的路径？哪些步骤是冗余的？
3. **提炼**：能提炼什么可复用的方法论？
4. **更新**：将改进写入规则系统或 discoveries.md

#### Scenario: 任务完成后自动复盘
- **WHEN** AI 完成一个 spec 的所有 task
- **THEN** AI 自动执行复盘，输出：
  - 做了什么（步骤回顾）
  - 改进空间（更快路径）
  - 可提炼方法论
  - 需要更新的规则/discoveries

### Requirement: 推理→搜索→验证三步法

AI 在开始搜索之前 SHALL 先推理可能的答案，再针对性搜索验证。

#### Scenario: 搜索前先推理
- **WHEN** AI 需要搜索代码来回答问题
- **THEN** AI 先从已知架构推断可能的答案，列出关键假设，再针对性搜索验证
