# Anchor 共享知识库 — 模块注册表

> 系统名称: Anchor
> 本注册表集中管理所有 shared/ 模块的元数据。AI 通过读取此文件了解有哪些模块可用。
> 增删改模块只需修改本文件 + 对应模块文件，AGENTS.md 永远不需要改。

## 读取顺序

| 优先级 | 模块 | 描述 | 读取时机 | 写入时机 |
|--------|------|------|---------|---------|
| P0 必读 | context.md | 项目核心上下文 | 每个新会话 | 项目重大变更时 |
| P1 推荐 | status.md | 当前状态和待办 | 每个新会话 | 每次会话结束时 |
| P2 按需 | discoveries.md | 重要发现和代码定位 | 需要相关知识时 | 发现关键信息时 |
| P2 按需 | decisions.md | 技术决策记录 | 需要理解决策时 | 做出重要决策时 |
| P2 按需 | rules.md | 协作规则 | 需要了解规则时 | 修改 rules/*.yaml 后运行引擎 |

## 写入格式约定

所有 shared/ 模块追加新条目时，统一使用以下格式：

### [YYYY-MM-DD HH:mm] 简短标题

**关键字1**: 值1
**关键字2**: 值2

详细描述...

---

**⚠️ 追加而非重写**: 向 shared/*.md 写入时，必须在文件末尾追加新条目，不得重写整个文件。重写会导致其他会话写入的条目丢失。只允许更新 status.md 中的结构性内容（如表格行），但不得删除其他会话写入的条目。

**会话日志格式**（写入 status.md 的"会话日志"区域）：

### [YYYY-MM-DD HH:mm] 会话 #N — 简短描述

**操作**: 本次会话做了什么
**观察**: 观察到什么现象或模式
**问题**: 遇到什么问题
**建议**: 对后续会话的建议

---

## 模块管理

- **新增模块**: 创建 `shared/新模块.md`（含自描述头部） + 在本表添加一行
- **删除模块**: 删除文件 + 从本表移除对应行
- **修改优先级**: 只改本表中的优先级列
- **修改格式**: 只改本文件中的"写入格式约定"章节

## 系统改名指引

如果需要更改系统名称（当前: Anchor），**只需修改以下 2 个文件**：

1. **AGENTS.md** — "Anchor 声明" 和 "Anchor 共享知识库"（2 处）
2. **本文件** — 标题行和 `> 系统名称:` 行（2 处）

**不要修改的文件**：
- `shared/*.md`（context/status/discoveries/decisions）— 已去品牌化，不含系统名
- `progress.txt` — 历史记录，应保持原名
- `.trae/specs/` — 设计决策历史，应保持原名

## 设计原则

以下原则是 Anchor 系统的设计约束，任何修改都应遵守。违反这些原则会导致系统维护性退化。

### 1. 去品牌化
- **描述**: shared/*.md 只描述功能，不含系统品牌名（如 "Anchor"）
- **理由**: 品牌名散布在所有文件中会导致改名时"牵一发动全身"
- **违反后果**: 改名时需要修改所有 shared/*.md 文件，而非只改 2 个文件

### 2. 集中定义
- **描述**: 系统名只在 AGENTS.md + _registry.md 中硬编码
- **理由**: 集中定义让改名只需改 2 个文件（共 4 处）
- **违反后果**: 系统名散布到其他文件后，改名需要全项目搜索替换

### 3. 历史不改
- **描述**: 历史文件（progress.txt, .trae/specs/）保持创建时的原始名称
- **理由**: 改名等于篡改历史，会让读者困惑时序
- **违反后果**: 读者无法判断某条记录是当时写的还是后来改的

### 4. AGENTS.md 只做路由
- **描述**: AGENTS.md 不存储具体内容，只指向 _registry.md
- **理由**: AGENTS.md 保持精简才能确保 AI 每次都完整读取
- **违反后果**: AGENTS.md 膨胀后 AI 可能跳过或忽略部分内容

### 5. 牵一发动全身
- **描述**: 增删模块不应影响其他文件，只改 _registry.md + 模块文件本身
- **理由**: 模块间解耦才能保证系统的可维护性
- **违反后果**: 新增模块需要修改多个文件，维护成本随模块数线性增长

## 移植清单

将 Anchor 系统移植到新项目时，需要复制以下文件和目录：

| 文件/目录 | 作用 | 必需 |
|-----------|------|------|
| `AGENTS.md` | Anchor 入口，AI 每次必读 | ✅ 必须 |
| `shared/` | 知识模块目录（含本文件） | ✅ 必须 |
| `shared/_registry.md` | 模块注册表 | ✅ 必须 |
| `shared/context.md` | 项目核心上下文 | ✅ 必须 |
| `shared/status.md` | 当前状态和待办 | ✅ 必须 |
| `shared/discoveries.md` | 重要发现 | ✅ 必须 |
| `shared/decisions.md` | 技术决策记录 | ✅ 必须 |
| `shared/rules.md` | 协作规则（自动生成） | ⚠️ 可选 |
| `rules/` | 规则定义（YAML 源文件） | ⚠️ 可选 |
| `scripts/rules-engine.ps1` | 规则引擎脚本 | ⚠️ 可选（无规则则不需要） |

**移植步骤**：
1. 复制上述文件到新项目根目录
2. 编辑 `AGENTS.md`，将项目特定信息替换为新项目的内容
3. 编辑 `shared/context.md`，更新项目核心上下文
4. 清空 `shared/status.md`、`shared/discoveries.md`、`shared/decisions.md` 中的旧条目（保留格式模板）
5. 如需规则系统，复制 `rules/` 和 `scripts/rules-engine.ps1`，运行 `powershell scripts/rules-engine.ps1 -Output shared/rules.md`
6. 如不需规则系统，从 _registry.md 的模块表中移除 `rules.md` 行，删除 `rules/` 目录
